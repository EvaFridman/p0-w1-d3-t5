# p0-w1-d3-t5 · Мини-сайт мероприятия (большая задача недели)

> Тип: большой синтез всех трёх дней · Репозиторий: один отдельный · Объёмнее обычных задач

## Контекст

Локальный IT-митап заказал простой сайт: рассказать о событии, показать программу и собрать
регистрацию. Бюджета на дизайнера и фронтендера нет — есть ты и чистый HTML. Дело за малым:
сверстать аккуратно, семантически и так, чтобы сайтом мог пользоваться вообще любой человек,
включая тех, кто ходит с клавиатуры и слушает страницу скринридером.

Это итоговая задача за три дня: ты прогоняешь руками сразу всё — структуру и семантику (день 1),
формы (день 2), доступность (день 3). Кода здесь заметно больше, чем в дневных задачах: три
связанные страницы с общим каркасом, а не один файл.

```
мини-сайт мероприятия — 3 страницы, общий каркас
┌────────────────────────────────────────────────────┐
│ <header> + <nav>  (Tab-доступная, skip-link)       │
│ ───────────────────────────────────────────        │
│ О событии   │  Программа   │  Регистрация          │
│  семантика  │  списки/     │  доступная форма      │
│  alt/имена  │  таблица     │  fieldset+aria        │
│ ───────────────────────────────────────────        │
│ <footer>  контакты, год (<time>)                   │
└────────────────────────────────────────────────────┘
одна разметка-эталон: валидна, доступна, без JS
изюминка: печатная версия + проверка контраста
```

## Что нужно сделать

Свёрстанный на чистом HTML мини-сайт из трёх страниц с общим каркасом (шапка с навигацией +
подвал повторяются на всех): `index.html` (о событии), `program.html` (программа), `register.html`
(регистрация). Без JS — на первой неделе его ещё нет. CSS не требуется (можно совсем без него);
вся ценность в разметке: она должна быть валидной, семантичной и доступной.

## Релизы

### Release 1 — общий каркас и навигация
Собрать одинаковую для всех трёх страниц шапку (`header` + `nav` со ссылками между страницами) и
подвал (`footer` с контактами и годом через `<time>`). Навигация — доступная с клавиатуры,
активный пункт помечен `aria-current="page"` на своей странице, в начале — skip-link к
основному содержимому. Все три страницы открываются и ссылаются друг на друга.

### Release 2 — страница «О событии»
`index.html`: семантический контент (заголовки правильной иерархии, `article`/`section`,
картинка площадки с осмысленным `alt`, врезка `aside` с датой/местом). Один `h1`, один `main`.
Прогнать через W3C-валидатор (навык из дня 1, t4) — ноль ошибок.

### Release 3 — страница «Программа»
`program.html`: расписание докладов. Подумать, что здесь семантически правильнее — список или
таблица; если таблица, сделать её доступной (`caption`, `th` с `scope`). Время докладов — через
`<time datetime="...">`. Структура читается скринридером как осмысленное расписание.

### Release 4 — страница «Регистрация»
`register.html`: доступная форма (синтез дня 2 и дня 3). Поля сгруппированы `fieldset`/`legend`,
каждое с `label`, сложные — с подсказкой через `aria-describedby`, зона ошибок с `aria-live`.
Нативная валидация (`required`, `type`, `pattern`), без поля «пол». Отправка нативная — `method`
и `action` на эхо-сервис. Форма полностью заполняется с клавиатуры.

### Release 5 — изюминка: печать и контраст (нужно погуглить)
Две вещи, которых на лекции не было — раскопать самой:
1. **Печатная версия.** Сделать так, чтобы при печати страницы (Cmd+P → сохранить в PDF)
   навигация и подвал-меню не печатались, а контент оставался читаемым. Это делается
   `@media print` в CSS — найти, как скрывать элементы только при печати (без JS).
2. **Контраст.** Если добавила хоть какие-то цвета — проверить пары текст/фон на соответствие
   WCAG AA (4.5:1) через онлайн-контраст-чекер. Если цветов нет — добавить минимальную цветовую
   тему и довести её до прохождения AA.
   
В `NOTES.md` записать: как работает `@media print`, какие пары проверяла на контраст и какие
значения получились.

## Критерии приёмки

* Три страницы с единым каркасом, ссылаются друг на друга, активный пункт помечен `aria-current`.
* Каждая страница: один `h1`, один `main`, валидна по W3C (ноль ошибок), есть skip-link.
* Весь сайт проходится только с клавиатуры; у всех картинок осмысленный `alt`, у интерактивного —
  доступные имена.
* Форма регистрации доступна, сгруппирована `fieldset`/`legend`, с нативной валидацией,
  отправляется без JS; поля «пол» нет.
* Программа оформлена семантически корректно (обоснован выбор список/таблица; таблица — с
  `caption`/`th scope`), время — через `<time>`.
* При печати скрыты навигация и подвал-меню; цветовые пары (если есть) проходят WCAG AA.
* Во всём проекте ни строки JavaScript.

## К сдаче

Уметь объяснить: почему на «Программе» выбрала список или таблицу; чем `aria-current` лучше, чем
просто подсветка активного пункта; как именно `@media print` отделяет экранную версию от
печатной.

## Подсказки

* Печать: ищи «CSS @media print», «hide element when printing», «print-friendly page CSS».
* Контраст: ищи «WCAG AA contrast checker», «contrast ratio 4.5:1».
* Тянет писать `<script>` — свернула не туда, вся задача решается HTML и CSS.
